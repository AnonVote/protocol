# Cryptographic Primitive Specifications

**Package:** [`@anonvote/crypto`](https://github.com/AnonVote/js)<br>
**Reference implementation:** `AnonVote/js/src/crypto.ts`<br>
**Status:** Normative for AnonVote/core, AnonVote/js, and audit tooling

This document specifies the five cryptographic primitives that define the
AnonVote privacy layer. It is written for implementers and reviewers: every
primitive below names its algorithm, parameters, input and output formats,
security property, and failure modes.

The reference implementation currently exports:

- `hashIdentifier(id: string): string`
- `generateToken(): string`
- `hashToken(token: string): string`
- `encryptVote(optionId: string, ballotKey: string): string`
- `decryptVote(payload: string, ballotKey: string): string`

These functions compose into AnonVote's structural unlinkability model:
identifiers prove eligibility, random tokens authorize one vote, token hashes
prevent token disclosure at rest, and authenticated encryption hides the chosen
option until tally time.

---

## Protocol Invariants

Any compatible implementation MUST preserve these invariants:

1. Raw voter identifiers MUST NOT be persisted after eligibility hashing.
2. Raw voter tokens MUST NOT be persisted after token issuance.
3. `hashIdentifier` and `hashToken` MUST remain independent domains. Token
   values are not normalized and are not mixed with identifier values.
4. Every vote encryption call MUST use a fresh IV generated by a CSPRNG.
5. A failed AES-GCM authentication check MUST abort decryption. The vote MUST
   NOT be silently mapped to a fallback option, blank option, or partial tally.
6. Ballot encryption keys MUST NOT appear in public API responses, audit events,
   Stellar data, Soroban contract state, or application logs.

## Dependency Diagram

```text
Admin uploads eligibility list
        |
        v
hashIdentifier(identifier)
        |
        v
store EligibilityEntry.identifierHash
        |
        v
Voter requests token with identifier
        |
        v
hashIdentifier(identifier) --> lookup EligibilityEntry
        |
        v
generateToken() --> raw token returned once to voter
        |
        v
hashToken(raw token) --> store VoterToken.tokenHash
        |
        v
Voter submits raw token + optionId
        |
        v
hashToken(raw token) --> lookup unused VoterToken
        |
        v
encryptVote(optionId, ballotKey)
        |
        v
store Vote.encryptedPayload and mark token used
        |
        v
Tally time
        |
        v
decryptVote(encryptedPayload, ballotKey)
        |
        v
aggregate plaintext optionId values into the ballot result
```

The diagram intentionally has no edge from `EligibilityEntry` to `Vote`. The
shared ballot context is sufficient for counting, but no cryptographic or
database value links a voter identifier to a selected option.

## `hashIdentifier(id: string): string`

**Reference:** `AnonVote/js/src/crypto.ts` export `hashIdentifier`

### Algorithm and Parameters

| Property | Requirement |
| --- | --- |
| Hash algorithm | SHA-256 |
| Input preprocessing | `id.trim().toLowerCase()` |
| Digest encoding | Lowercase hexadecimal |
| Output length | 64 hex characters / 32 bytes |
| Deterministic | Yes |

### Input Format and Constraints

`id` is the voter identifier supplied by an administrator or voter. Common
examples are email addresses, employee IDs, student IDs, or organization member
IDs. The primitive accepts a string; caller layers SHOULD reject empty
identifiers and SHOULD use one canonical identifier scheme per eligibility list.

Normalization is required before hashing:

```text
normalizedIdentifier = trim(id).toLowerCase()
identifierHash       = SHA-256(normalizedIdentifier)
```

Whitespace trimming and lowercasing exist so accidental casing differences do
not split one real voter into multiple eligibility records.

### Output Format and Guarantees

The output is a 64-character lowercase hex string suitable for storage as
`EligibilityEntry.identifierHash`. The raw identifier MUST NOT be stored beside
the hash.

### Security Property

The enforced property is preimage resistance: a database reader should not be
able to recover the original identifier from the stored hash. SHA-256 also makes
identifier comparison deterministic without persisting the raw value.

### Failure Modes

- If normalization is omitted, `Alice@example.com` and `alice@example.com` can be
  treated as different voters, allowing duplicate token issuance or false
  ineligibility.
- If a reversible transform or weak hash is used, a database compromise can
  expose the eligibility list.
- If raw identifiers are logged or stored next to the hash, the privacy
  guarantee is void regardless of the hash strength.

## `generateToken(): string`

**Reference:** `AnonVote/js/src/crypto.ts` export `generateToken`

### Algorithm and Parameters

| Property | Requirement |
| --- | --- |
| Random source | Cryptographically secure PRNG |
| Reference source | Node.js `crypto.randomBytes(32)` |
| Entropy | 256 bits |
| Encoding | Lowercase hexadecimal |
| Output length | 64 hex characters / 32 bytes |
| Deterministic | No |

### Input Format and Constraints

`generateToken` takes no input. It MUST read 32 bytes from a CSPRNG every time it
is called.

### Output Format and Guarantees

The output is a raw voter token delivered to the voter exactly once. It is
probabilistically unique: a collision is not mathematically impossible, but with
256 bits of entropy it is infeasible in practice. Storage layers MAY still
enforce a unique constraint on `VoterToken.tokenHash`; a collision or duplicate
hash MUST be treated as an issuance failure and retried or escalated.

### Security Property

The enforced property is unpredictability. A token holder can prove possession
of voting authority, but an attacker cannot guess a valid token by observing
other tokens or token hashes.

### Failure Modes

- `Math.random()`, timestamps, counters, UUID versions without sufficient
  entropy, or any non-CSPRNG source are unacceptable because outputs can become
  guessable.
- Shorter tokens reduce brute-force cost and can make online guessing viable.
- Persisting the raw token gives any database reader immediate voting authority.
  Only `hashToken(rawToken)` may be persisted.

## `hashToken(token: string): string`

**Reference:** `AnonVote/js/src/crypto.ts` export `hashToken`

### Algorithm and Parameters

| Property | Requirement |
| --- | --- |
| Hash algorithm | SHA-256 |
| Input preprocessing | None |
| Digest encoding | Lowercase hexadecimal |
| Output length | 64 hex characters / 32 bytes |
| Deterministic | Yes |

### Input Format and Constraints

`token` is the exact raw token returned by `generateToken`. It MUST NOT be
trimmed, lowercased, decoded, parsed, or otherwise normalized before hashing.
The token domain is intentionally case-sensitive and byte-sensitive.

### Output Format and Guarantees

The output is a 64-character lowercase hex string suitable for storage as
`VoterToken.tokenHash`. A later vote submission recomputes the hash from the raw
token and looks up the stored value.

### Security Property

The enforced property is bearer-token secrecy at rest. If the database is
compromised, an attacker sees token hashes but not raw tokens. Because the raw
token has 256 bits of CSPRNG entropy, offline preimage search is infeasible.

### Two-Step Design Rationale

Token generation and token hashing are separate operations because the raw token
has two different lifetimes:

1. The raw token is given to the voter once.
2. The stored token hash persists until the token is used, expires, or is
   reissued.

Separating `generateToken` from `hashToken` makes that boundary auditable.
Application code can be reviewed for "raw token returned, hash stored" without
also inspecting randomness generation details.

### Failure Modes

- If the raw token is stored, copied to logs, or embedded in audit events, token
  possession can be stolen from storage.
- If token hashing normalizes input, a transformed token can become valid even
  though the exact issued token was never presented.
- If token hashing is replaced with identifier hashing, the token and identity
  domains can become linkable.

## `encryptVote(optionId: string, ballotKey: string): string`

**Reference:** `AnonVote/js/src/crypto.ts` export `encryptVote`

### Algorithm and Parameters

| Property | Requirement |
| --- | --- |
| Encryption algorithm | AES-256-GCM |
| Key length | 32 bytes |
| Key input encoding | 64-character hex string |
| IV length | 12 bytes / 96 bits |
| IV source | Fresh CSPRNG bytes per call |
| Auth tag | GCM authentication tag, 16 bytes in the reference implementation |
| Plaintext encoding | UTF-8 `optionId` string |

### Input Format and Constraints

`optionId` is the ballot option identifier being selected. It MUST be a valid
option for the target ballot before encryption is called. It MUST NOT contain a
voter identifier, raw token, token hash, IP address, user agent, or other voter
metadata.

`ballotKey` is the 64-character hex representation of a 32-byte ballot
encryption key. The reference implementation validates key length before
constructing the cipher. Compatible implementations SHOULD also validate that
the string is valid hex.

### Output Format and Guarantees

The output is a colon-separated string:

```text
base64(iv):base64(authTag):base64(ciphertext)
```

This order is normative because `decryptVote` parses the payload as
`iv:authTag:ciphertext`. The ciphertext is not deterministic: encrypting the
same option twice under the same key MUST produce different payloads because the
IV is freshly generated.

### Security Property

AES-256-GCM provides authenticated encryption. The selected option is
confidential while stored, and any modification to IV, auth tag, or ciphertext is
detected during decryption.

GCM is used instead of CBC because CBC encryption alone does not authenticate
the ciphertext. CBC deployments require a separate MAC and careful ordering to
avoid padding-oracle and malleability issues. GCM provides confidentiality and
integrity in one AEAD mode and avoids padding.

### Failure Modes

- Reusing an IV with the same key can compromise confidentiality and integrity.
  IV generation MUST be fresh for every call.
- Using one global key for all ballots increases blast radius. See key
  management below for the protocol requirement.
- Encrypting voter metadata together with `optionId` weakens unlinkability.
- If the auth tag is discarded or stored separately from the payload, later
  decryption cannot reliably detect tampering.

## `decryptVote(payload: string, ballotKey: string): string`

**Reference:** `AnonVote/js/src/crypto.ts` export `decryptVote`

### Algorithm and Parameters

| Property | Requirement |
| --- | --- |
| Decryption algorithm | AES-256-GCM |
| Key length | 32 bytes |
| Key input encoding | 64-character hex string |
| Input format | `base64(iv):base64(authTag):base64(ciphertext)` |
| Output | UTF-8 `optionId` string |
| Auth verification | Required before plaintext is accepted |

### Input Format and Constraints

`payload` MUST contain exactly three colon-separated base64 segments in this
order: IV, auth tag, ciphertext. `ballotKey` MUST be the same key used by
`encryptVote` for that ballot.

### Output Format and Guarantees

On success, the output is the original `optionId`. The caller MUST validate that
the option still belongs to the ballot being tallied before counting it.

### Security Property

The enforced property is tamper detection. A successful decrypt means the
payload was produced under the ballot key and has not been modified. It does not
prove who submitted the vote.

### Failure Modes

- If payload parsing fails, the stored vote is malformed and tallying SHOULD
  abort with an audit-visible error.
- If auth tag verification fails, the payload, tag, IV, or key is wrong. In the
  voting context this means the vote record cannot be trusted. The tally MUST
  NOT silently count it as blank, ignore it without alerting, or substitute a
  default option.
- Silent failure is worse than a thrown error because it can hide database
  tampering or key mismatch and produce a plausible but incorrect tally.

## Key Management

The `@anonvote/crypto` functions accept `ballotKey` as an argument, but key
generation and storage are the responsibility of AnonVote/core. The protocol
requires per-ballot keys.

### Generation

For every ballot, core MUST generate a new 32-byte encryption key using a CSPRNG
at ballot creation time:

```text
ballotKey = CSPRNG(32 bytes) -> 64-character hex string
```

Keys MUST NOT be derived from ballot IDs, organization IDs, passwords, token
hashes, timestamps, or any public value.

### Storage

The preferred storage location is a secrets manager or KMS. If deployment
constraints require database-backed storage, the key MUST be encrypted at rest
under an application master key that is not stored in the same database. The
public `Ballot` record MAY store a key identifier, but it MUST NOT store the
plaintext ballot key.

### Use

- Vote submission loads the ballot key, calls `encryptVote(optionId, ballotKey)`,
  and stores only the encrypted payload.
- Result tally loads the same ballot key, calls `decryptVote(payload,
  ballotKey)` for each vote, and aggregates the recovered option IDs.
- Audit events and on-chain records MUST include counts and hashes only; they
  MUST NOT include the ballot key or plaintext option IDs unless the result has
  intentionally been published.

### Rotation and Retention

Per-ballot keys limit blast radius. If one key is exposed, only that ballot's
stored vote payloads are decryptable. Closed ballot keys SHOULD be retained only
as long as the deployment requires recounts or audits. Destroying a key makes
future decryption impossible, so retention policy must be explicit before key
destruction.

## Threat Model Boundaries

The crypto layer protects against:

- Database readers linking raw identifiers, raw tokens, and encrypted votes.
- Offline guessing of raw tokens from token hashes.
- Stored vote tampering that modifies ciphertext, IV, or auth tag.
- Accidental duplicate token issuance caused by identifier casing or whitespace
  differences.

The crypto layer does not protect against:

- **Traffic analysis:** a network observer may correlate token request timing
  and vote submission timing. Transport security and operational controls are
  required outside this primitive layer.
- **Coercion:** a voter can be forced to reveal a token or vote a certain way.
  This protocol does not provide coercion resistance.
- **Admin key compromise:** if a ballot key is exposed, stored votes for that
  ballot can be decrypted. Per-ballot keys reduce blast radius but do not make a
  compromised key safe.
- **Malicious eligibility administration:** an admin can include, exclude, or
  weight voters incorrectly before cryptographic processing begins.
- **Endpoint compromise:** if application code is modified before primitives are
  called, it can log raw identifiers, raw tokens, or plaintext votes.
- **Availability attacks:** cryptography does not prevent denial-of-service or
  API downtime.

Auditors should therefore inspect both primitive correctness and operational
controls: logging, token delivery, rate limiting, key storage, deployment
secrets, and tally error handling.

---

## Reference Checklist

When reviewing an implementation, verify:

- `hashIdentifier` uses SHA-256 over `trim().toLowerCase()` output.
- `generateToken` uses a CSPRNG for exactly 32 random bytes.
- `hashToken` hashes the exact raw token without normalization.
- `encryptVote` uses AES-256-GCM with a fresh 12-byte IV for every call.
- `encryptVote` returns `base64(iv):base64(authTag):base64(ciphertext)`.
- `decryptVote` verifies the auth tag and throws on malformed or tampered
  payloads.
- Core uses a per-ballot 32-byte key and never stores it in public data, logs,
  audit events, Stellar data, or Soroban state.
